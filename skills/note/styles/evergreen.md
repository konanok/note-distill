# Style: evergreen（Evergreen Notes）

## 定位

面向**知识积累和复用**。目标不是复现一次操作，而是沉淀一个可以持续生长的观点或洞见。适合需要构建个人知识体系的用户。

## 写作哲学

受 Andy Matuschak 的 Evergreen Notes 启发：

- **原子化**：每篇只表达一个观点，不是一个操作流程
- **命题式标题**：标题本身就是一个完整论断，读者看标题就知道核心观点
- **面向复用**：通过 wikilinks 与其他笔记连接，随时间积累形成知识网络
- **持续迭代**：笔记可以被修改和完善，不是一次性记录

## 对模板的覆盖规则

覆盖 `depth-template.md`，结构如下：

```markdown
---
{frontmatter}
---

# {完整命题句标题}

## 核心观点

{2-4 句话阐明这个命题的核心主张。直接、明确，不铺垫背景。}

## 论据与支撑

{支撑核心观点的证据、案例、原理。可分小节。}

### {论据1标题}

{...}

### {论据2标题}（可选）

{...}

## 边界与反例

- **何时不成立**: {这个观点在什么条件下不适用}
- **反例或例外**: {已知的反例}

## 关联条目

- [[相关观点或概念]] — {一句话说明关联关系}
- [[...]]
- 来源：{文章/文档/对话 URL 或描述}
```

覆盖说明：
- 去掉「背景与问题」（evergreen 不以问题为起点，以观点为起点）
- 去掉「解决方案」和「关键代码/命令」（观点笔记不是操作手册）
- 去掉「验证证据」（改为「论据与支撑」，逻辑不同）
- 保留「备选方案与取舍」的精神，改为「边界与反例」
- 强化「关联条目」，这是 evergreen 的核心

## 标题规范

**强制格式**：完整命题句，表达一个可以被认同或反驳的主张。

示例：
- `Git rebase 比 merge 更适合维护线性提交历史`
- `过早抽象比重复代码的危害更大`
- `好的错误信息应该说明原因而不只是现象`

**禁止**：
- ❌ 动词开头的操作描述（那是 technical 风格）
- ❌ 模糊名词短语（如「Git 使用技巧」）
- ❌ 问句（如「什么时候用 rebase？」）

## frontmatter

在 adapter 默认 frontmatter 基础上，强制添加：

```yaml
tags: [evergreen, <domain-tag>]
status: growing              # seed → growing → evergreen，初始为 growing
```

## 文件存放

覆盖 adapter 的默认路径，存入 `{vault}/Evergreen/` 目录。

## wikilinks

Evergreen 的核心机制，**必须认真对待**：

- 每篇要求 5-10 个 wikilinks
- 链接对象：其他观点笔记、核心概念、相关技术术语
- 每个 wikilink 后加一句关联说明（在「关联条目」section 中）
- 即使 vault 里暂时没有对应笔记，也应写出 wikilinks（Obsidian 会自动创建占位）

## 适用模式

**仅适用于 deep 模式**。若用户指定 `--style evergreen` 同时指定 `quick`，subagent 应忽略 quick，强制使用 deep。

若 MODE=auto 且 style=evergreen，直接选 deep，不再判断启发式规则。
